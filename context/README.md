# Context — what is true *right now*

The **soul** is who the agent is; it changes rarely. **Context** is what is true
*this session*; it is rebuilt every time the agent starts. The two are
deliberately separate: you author your character once, but your branch, your
repos, your identity, and your live inventory are different every morning.

## What goes in context

Anything the agent needs to know about the present moment that it shouldn't have
to discover by hand:

| Kind | Examples |
|------|----------|
| **Identity** | who the user is — name, role, how to address them — resolved from a profile, *not* from the harness account |
| **Environment** | working directory, OS, shell, configured repo paths, current git branch |
| **Live inventory** | the list of services / apps / domains that exist *today*, generated by scanning the repos |
| **Tooling state** | which MCP servers are connected, which CI is configured, which AWS profiles exist |
| **Time** | the actual current date and time — so "latest", "today", and "this week" mean something |

## Why it's injected, not authored

If you hard-code "we have 30 backend services" into the soul, the soul lies the
day someone adds the 31st. Context is **generated at startup** by inspecting the
real world — listing directories, reading the profile, querying which servers
connected — and spliced into the system prompt fresh each time. The agent never
reasons from a stale snapshot because there is no snapshot to go stale.

```
startup:
  read profile          → identity block
  scan repo paths       → live inventory
  check connected MCPs   → tooling state
  read clock            → current date/time
  ──────────────────────────────────────────
  splice all of it into the system prompt, below the soul
```

## The golden rule for the agent

**Context is authoritative over memory.** A recalled memory reflects what was
true when it was written; the context block reflects what is true now. When they
disagree — a memory names a service that the live inventory no longer lists — the
context wins, and the memory is stale and should be re-verified or deleted.

## The identity trap

The harness running your agent has its own notion of "the user" — the account it
authenticated with, an auto-memory email, an OS username. **That is the operator
of the harness, not necessarily the person speaking.** Resolve identity from
*your own* authoritative source (a profile file you control) and tell the agent,
in the soul, to ignore the harness's guess. Getting this wrong means the agent
addresses the user as the wrong person — a small bug with an outsized effect on
the illusion of a companion who knows them.

See `environment.example.md` for the shape of an injected context block.
